## 2. System Overview

### 2.1 System Architecture
The `vgp` application is a command-line tool. The primary components under consideration are:
*   **`main.c`**: Acts as the controller. It parses command-line arguments, manages file I/O for the primary input (Valgrind log) and output, and invokes the parsing engine.
*   **`vgp.c`**: Implements the core parsing engine. It takes Valgrind log data as input, processes it to identify errors and summaries, and returns structured data.

The typical flow involves `main.c` receiving a Valgrind log file path (and options) from the user, passing this to `vgp.c` for parsing, and then `main.c` formatting and presenting the results from `vgp.c` to the standard output or a specified output file.

### 2.2 Design Goals and Constraints
*   **Accuracy:** Correctly parse and interpret Valgrind log information.
*   **Performance:** Efficiently process potentially large Valgrind log files.
*   **Usability:** Provide clear and useful output. Support relevant command-line options for controlling behavior (e.g., verbosity, input/output).
*   **Maintainability:** Code should be well-structured, commented, and modular.
*   **Portability:** Primarily target Linux environments (as suggested by Makefile paths like `/usr/local/bin`).
*   **Input Format:** Assume standard Valgrind text output (unless XML output is explicitly targeted).

## 4. Detailed Design for `/home/john/Projects/Valgrind_Parser/vgp.c`

### 4.1 Purpose and Responsibilities
`/home/john/Projects/Valgrind_Parser/vgp.c` implements the core logic for parsing Valgrind output logs. Its responsibilities include:
*   Reading and interpreting the content of a Valgrind log.
*   Identifying different sections of the log (e.g., error summaries, specific error reports, leak summaries).
*   Extracting detailed information such as error types, memory addresses, sizes, call stacks, and leak categorizations.
*   Storing this extracted information in well-defined data structures.
*   Providing an API for `main.c` (or other modules) to initiate parsing and retrieve the parsed data.

#### 4.3.3 Parsing Strategy/Algorithm
*   The primary parsing function (`vgp_parse_log_stream`) will read the Valgrind log line by line.
*   A state machine (`VgpParseState` enum or struct) might be used to keep track of the current context within the log (e.g., `IDLE`, `IN_ERROR_BLOCK`, `IN_LEAK_SUMMARY`, `PARSING_CALLSTACK`).
*   Regular expressions (if a regex library is used) or `strstr`/`sscanf` combinations will be employed to identify key patterns and extract data.
    *   Example patterns:
        *   `==<PID>== Invalid read of size ...`
        *   `==<PID>==    at 0x...: function_name (source_file:line_number)`
        *   `==<PID>==    by 0x...: ...`
        *   `==<PID>== LEAK SUMMARY:`
        *   `==<PID>==    definitely lost: X bytes in Y blocks`
*   Dynamic memory allocation will be used for storing variable-length data like strings, lists of errors, and call stacks. Careful memory management is crucial.
